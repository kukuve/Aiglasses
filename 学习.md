# 前端注册/登录数据与 @admin 同步问题排查与修复记录

## 1. 问题背景

现象：用户在前端 `User_UI` 完成注册/登录后，`@admin` 管理端未看到对应用户数据或登录状态更新，表现为“数据未同步”。

目标：确保注册和登录产生的用户数据可以实时、准确地在 `@admin` 可视化页面展示。

---

## 2. 系统链路梳理（注册/登录）

### 2.1 前端注册链路

1. `User_UI/pages/register/register.js` 调用：
   - `wx.cloud.callFunction({ name: 'userAuth', data: { action: 'register', phone, code, password } })`
2. `admin/cloudfunctions/userAuth/index.js` 的 `register()`：
   - 校验手机号、验证码、密码
   - 写入 `users` 集合
   - 返回 `sessionToken`、`sessionExpiresAt`、`userInfo`
3. 前端 `User_UI/app.js` + `User_UI/utils/auth.js`：
   - 保存会话到本地 storage 与 globalData

### 2.2 前端登录链路

1. `User_UI/pages/login/login.js` 调用：
   - `wx.cloud.callFunction({ name: 'userAuth', data: { action: 'login', account, password } })`
2. `admin/cloudfunctions/userAuth/index.js` 的 `login()`：
   - 校验账号密码
   - 更新会话信息（token、过期时间、最近登录时间）
3. 前端保存会话并跳转业务页

### 2.3 管理端展示链路（修复前）

`admin/miniprogram/pages/dashboard/index.js` 原先调用的是 `vocabApi`（词汇统计/词条列表），并非用户数据接口。

结论：即使用户注册/登录成功写入了 `users`，管理端也不会展示用户表，造成“未同步”的直观问题。

---

## 3. 复现与确认

### 3.1 可复现表现

- 用户端完成注册/登录后，管理端 dashboard 显示的是“词汇数据看板”，不是“用户表”。
- 页面数据源来自 `vocabApi`，不包含 `users` 集合数据。

### 3.2 前端请求检查（代码级）

从代码确认：

- 注册请求
  - 函数名：`userAuth`
  - action：`register`
  - payload：`phone/code/password`
- 登录请求
  - 函数名：`userAuth`
  - action：`login`
  - payload：`account/password`

前端代码存在错误兜底（`Auth.getFriendlyError`），可提示云函数未部署等异常。

### 3.3 环境一致性检查

两端均初始化到同一云环境：

- `User_UI/app.js` -> `env: cloud1-d9gof7sc438491e13`
- `admin/miniprogram/app.js` -> `env: cloud1-d9gof7sc438491e13`

说明：非“不同云环境导致的数据隔离”问题。

---

## 4. 根因分析

根因 1（核心）：  
`@admin` 管理端 dashboard 的查询逻辑接在词汇 API（`vocabApi`）上，而不是用户 API（`userAuth` + `users` 集合）。

根因 2（次要）：  
用户数据虽然写入了基础字段，但缺少管理可视化常用的稳定统计字段（毫秒时间戳、登录计数）会让“实时登录态变化”不易稳定展示和统计。

---

## 5.2 管理端页面：`admin/miniprogram/pages/dashboard/*`

## 6. 数据库持久化检查要点

修复后注册/登录对应写入逻辑：

- 注册后应存在：
  - `phone/account/status`
  - `createdAt + createdAtMs`
  - `lastLoginAt + lastLoginAtMs`
  - `sessionToken/sessionExpiresAt`
  - `loginCount = 1`

- 登录后应更新：
  - `sessionToken/sessionExpiresAt`
  - `lastLoginAt + lastLoginAtMs`
  - `loginCount += 1`

## 9. 本次修复结论

- 问题已定位：管理端数据源接错（词汇链路代替用户链路）是不同步主因。
- 已修复：管理端改为用户 API，后端补充用户统计与行为字段记录。
- 结果：用户注册/登录数据可在 `@admin` 实时可视化展示，满足“同步到管理系统”的核心要求。

补充说明（稳定性）：

- `userStats` 已改为数据库 `count` 条件统计（`lastLoginAtMs >= 时间点`），避免“仅扫描前 1000 条”导致的大数据量统计偏差。
- `loginCount` 更新改为显式数值写入，减少历史老数据字段类型不一致时的递增失败风险。

## 10. 新增：后端记录密码（受控查看）

根据需求，后端在用户注册时新增保存一份可解密密文：

- 字段：`passwordRawEncrypted`
- 时间：`passwordRawUpdatedAtMs`

实现方式：

- 使用 `aes-256-cbc` 加密后入库（非明文直接存储）
- 密钥来自环境变量：`PASSWORD_ENCRYPT_SECRET`

仅超级管理员可查看：

- 新增云函数动作：`action: 'getUserSensitive'`
- 入参：
  - `userId`
  - `adminKey`
- 鉴权：
  - 需要与环境变量 `SUPER_ADMIN_KEY` 一致
- 返回：
  - `passwordPlain`（解密后的密码）

> 安全建议：  
> 生产环境务必设置高强度 `PASSWORD_ENCRYPT_SECRET` 和 `SUPER_ADMIN_KEY`，并限制此接口仅在内网管理端可调用。

## 14. 超级管理员查询补充修复

问题现象：

- 部分用户在超级管理员页查询时“明文密码”为 `--`。

原因：

- 历史用户创建时未写入 `passwordRawEncrypted`；
- 历史数据无法由哈希反推明文。

修复：

1. 登录成功后自动补存密文
   - 在 `login` 流程中使用本次登录输入密码回写：
   - `passwordRawEncrypted`、`passwordRawUpdatedAtMs`

2. 超管查询支持手机号
   - `getUserSensitive` 支持 `userId` 或 `phone` 任一查询
   - 超级管理员页面新增手机号输入

说明：

- 若某历史账号从未在修复后成功登录过，明文密码仍为空；
- 该账号在修复后登录一次后即可被超级管理员查询到明文密码。

## 15. 手机号查询“用户不存在”问题的实现步骤

问题现象：

- 超级管理员页输入手机号后返回“用户不存在”。

排查结论：

- 历史用户数据存在格式不统一问题，常见为：
  - `phoneHash` 缺失（老数据）
  - `phone` 存的是 `138xxxx`、`+86xxxx` 或 `86xxxx` 的不同格式
- 早期查询仅按 `phoneHash` 命中，导致老数据被漏查。

实施步骤：

1. 增加手机号候选集生成函数
   - 在 `userAuth` 新增 `getPhoneQueryCandidates(input)`：
     - 标准化手机号（`normalizePhoneNumber`）
     - 生成 `phoneHash`
     - 生成多种明文候选（`+86...`、`86...`、`11位手机号`）

2. 增加统一手机号查人函数
   - 新增 `findActiveUserByPhone(input)`：
     - 先查 `phoneHash`
     - 再按明文候选逐个查 `phone`
     - 仅返回 `status: 'active'` 用户

3. 统一替换关键调用点
   - `findUserByAccount`：手机号登录支持历史格式
   - `getUserSensitive`：超管手机号查询支持历史格式
   - `bindAccountByPhone`：管理员手机号绑定支持历史格式
   - `retrieveAccountByPhone`：手机号找回支持历史格式

4. 修复 userId 干扰手机号查询
   - `getUserSensitive` 调整为：
     - 先按 `userId` 查
     - 若未命中且提供了手机号，自动回退按手机号查

5. 校验结果
   - `ReadLints` 无新增报错
   - 查询链路兼容新老手机号格式

部署与验证：

1. 重新部署 `admin/cloudfunctions/userAuth`
2. 超级管理员页尽量仅输入手机号（`userId` 可留空）测试
3. 若仍异常，去云函数日志看 `getUserSensitive` 入参和分支命中情况
